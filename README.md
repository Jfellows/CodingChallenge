# Azure Cloud Platform Engineer – Technical Challenge

This repository contains the infrastructure and application source code to deploy a completely serverless Azure Function App using the **Windows Consumption** plan.

## Overview
- **Infrastructure**: Provisioned dynamically using Hashicorp Terraform via the `azurerm` provider. 
- **Application Payload**: An HTTP-triggered .NET 10.0 Azure Function using the Isolated Worker Model.
- **CI/CD**: Fully automated deployment available via GitHub Actions using modern Azure Workload Identity Federation (OIDC).

---

## ✅ Prerequisites
Ensure you have the following CLI tooling installed before proceeding:
- **[Azure CLI](https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)**
- **[Terraform](https://developer.hashicorp.com/terraform/downloads)** `>= 1.0.0`
- **[.NET SDK](https://dotnet.microsoft.com/download)** (Required for compilation)
- **[Azure Functions Core Tools](https://learn.microsoft.com/en-us/azure/azure-functions/functions-run-local)** (Simplifies local testing and deployment)

### 📌 A Note on State Management
Per the technical challenge parameters, this deployment deliberately relies on **Local State**. No remote backend configurations (like Azure Blob Storage) have been hardcoded. 

When you run `terraform apply`, Hashicorp Terraform will generate a `terraform.tfstate` file strictly on your local disk. 

*(Optional)* If you wish to explicitly enforce local state handling in your configuration, you can declare the `local` backend explicitly in your `provider.tf` file block:
```hcl
terraform {
  backend "local" {
    path = "terraform.tfstate"
  }
}
```

### 🤔 Architectural Assumptions & Access Model (Bonus Challenge)
The original challenge parameters outlined a hard constraint: `"Must use the Consumption (Free) plan"`. However, the Bonus Challenge requested configuring a **Private Endpoint** against the Data Plane. 

A fundamental constraint within Microsoft Azure is that standard Consumption Plans (the `Y1` SKU) **do not support Virtual Network Integration or Private Endpoints**. 

To satisfy the Bonus Challenge without violating the integrity of Terraform's infrastructure automation, we deliberately upgraded the App Service Plan in our configurations to the **Elastic Premium (`EP1`)** SKU. 
- **Why not Flex Consumption (`FC1`)?** While Azure recently introduced Flex Consumption as the successor serverless model (which *does* support private endpoints), integrating it via Terraform currently requires a highly rigid, uniquely scoped resource block (`azurerm_function_app_flex_consumption`) introduced in later `azurerm` v4 provider versions that drastically alters dependency schemas. Sticking with `EP1` ensures our `.NET 10.0` Windows implementation code functions impeccably out-of-the-box using the standard Function App API surface while fulfilling all Private Networking conditions gracefully!

## ✅ CLI Deployment Steps

### 1. Authenticate to Azure
Login to your Azure Account.
```bash
az login
```
*(If you have multiple tenants, use `az login --tenant <TENANT_ID>`)*

### 2. Initialize and Apply Terraform
Navigate to the `terraform` directory, pull down the necessary providers, and provision the serverless plan and storage dependencies.
```bash
cd terraform
terraform init
terraform plan --out tf.plan
terraform apply "tf.plan"
```
*(Wait a couple of minutes while the Azure infrastructure is dynamically spun up. Note the `function_app_name` that is outputted at the end!)*

### 3. Deploy the Function Code
Once infrastructure completes, you can push the raw .NET function payload straight to your Azure host.

**Deploy using Azure Functions Core Tools:**
```bash
cd ../function/http
func azure functionapp publish <YOUR_FUNCTION_APP_NAME> --dotnet-isolated
```

## 🚀 GitHub Actions CI/CD Pipeline

If you prefer to deploy application code automatically without using the CLI logic above, you can execute the included GitHub Actions workflow!

1. Create an **App Registration** and configure a **Federated Credential (OIDC)** pointing to your GitHub repo.
2. Under your Subscription's **Access Control (IAM)**, grant that new App Registration the **Contributor** role.
3. In your repo's GitHub Settings > **Secrets and Variables > Actions**, define these three Repository Secrets:
   - `AZURE_CLIENT_ID`
   - `AZURE_TENANT_ID`
   - `AZURE_SUBSCRIPTION_ID`
4. Deploy your `terraform` infrastructure manually first (since state is currently kept local). Take the exact Function App Name generated by Terraform and update the `FUNCTION_APP_NAME` variable inside `.github/workflows/deploy.yml`.
5. Navigate to the **Actions** tab on GitHub, select **Deploy Function App**, and click **Run Workflow**!

## ✅ Validation

You can definitively validate that the application works successfully (returning our target `Hello World` condition) without altering application logic or changing `AuthorizationLevel.Function`.

First, retrieve the securely generated Function key from the Azure host via the CLI:
```bash
az functionapp keys list -g funcapp-dev-rg -n <YOUR_FUNCTION_APP_NAME>
```

Take the `default` key returned in the JSON block, and append it to your request URL via the `?code=` parameter exactly like this:

**Using Curl (Bash):**
```bash
curl -X GET "https://<YOUR_FUNCTION_APP_NAME>.azurewebsites.net/api/httpget?code=YOUR_DEFAULT_KEY"
```

**Using PowerShell:**
```powershell
Invoke-RestMethod -Uri "https://<YOUR_FUNCTION_APP_NAME>.azurewebsites.net/api/httpget?code=YOUR_DEFAULT_KEY" -Method GET
```

**Expected Response String:**
```text
Hello, World.
```

## ✅ Cleanup / Teardown
To safely wipe out the provisioned Azure footprints and destroy the billing infrastructure, navigate back to the `terraform` directory and execute:
```bash
cd terraform
terraform destroy -auto-approve
```
